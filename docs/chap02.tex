\chapter{系统设计}
\label{cha:system_design}

\section{设计目标}
\label{sec:design_goals}

OpsEvolver 的设计目标不是构建一个只能在固定数据集上完成根因分类的模型，而是构建一个能够进入真实云原生环境、借助工具链与环境交互、并通过演练持续学习的原型系统。围绕这一目标，系统既要在告警出现后基于有限上下文组织检查路径，自主调用日志、指标、链路和命令行工具完成根因定位，也要在长期运行中不断吸收特定业务环境里的拓扑、服务命名和故障经验，使诊断决策不再完全依赖通用模型先验。与此同时，为了避免智能体执行不受约束的危险操作，故障注入动作必须被限制在可声明、可回滚、可追溯的受控集合中，而演练产出的知识也需要被系统化保存，以支撑后续评测与复用。

从系统工程角度看，上述需求意味着 OpsEvolver 既要具备典型 Agent 的“推理+工具调用”能力，也要具备运维系统所要求的“状态持久化+历史回放+知识追踪”特征。因此，OpsEvolver 采用“工作流驱动、多智能体协作、知识外置存储”的总体设计思路。

\section{方法论原则}
\label{sec:methodology_principles}

\subsection{主动学习：通过故障演练制造高价值样本}

传统告警驱动系统只能被动等待故障发生，样本稀缺且分布高度受生产流量影响。OpsEvolver 则通过混沌工程主动向环境施加扰动，将“系统可能出问题的地方”转化为可观测、可复盘、可沉淀的离线样本。对 SRE 智能体而言，主动演练的意义不在于单次注入本身，而在于把隐含的运维经验转化为结构化知识：哪些故障能够产生明显告警、哪些故障只在上游服务中体现、哪些故障会诱发误导性日志，都会在演练过程中逐步暴露出来。

进一步地，当前系统还显式记录“无告警注入”集合。也就是说，当某一类故障在当前负载和采样条件下没有形成可观测异常时，系统不会简单地把该轮实验视为失败，而是把它作为一类特殊知识保存下来，用于约束后续探索空间。这使主动学习过程不只是积累正样本，也在持续学习“哪些扰动在当前环境下不值得重复尝试”。

\subsection{可观测性感知：同时学习有效区域与无效区域}

传统故障演练往往只关心“有没有成功把故障注入进去”，但对智能运维系统而言，更关键的问题是“该故障是否真的在给定观测栈与工作负载下产生了可学习信号”。如果一次注入没有触发任何指标异常、链路退化或错误日志，那么它对系统诊断能力的提升就十分有限。基于这一认识，OpsEvolver 在方法论上把可观测性边界也纳入学习目标，将无告警注入视作一种负样本，而不是简单丢弃。

这样做的直接收益很明显。一方面，系统能够显式识别特定实验设置下的“不可观测区域”，避免反复消耗在短时间内难以产生告警的故障组合上；另一方面，后续实验分析也不再只统计知识条目和成功案例，而能同时刻画哪些故障类型受限于负载、采样窗口或业务路径，尚未形成稳定异常信号。换言之，OpsEvolver 学到的不仅是“什么故障会导致什么症状”，也包括“什么故障在既定环境里暂时学不到”。

\subsection{迭代优化上下文：持续更新诊断提示与知识库}

离线演化的核心价值在于它为智能体提供了一个持续学习的机制。每轮演练结束后，系统不仅会把诊断轨迹和评估结果转化为结构化知识条目，还会把这些条目组织成规则库，并在后续诊断过程中作为提示输入。这一机制把“知识积累”与“诊断优化”紧密结合起来：每次演练不仅是一次样本采集，更是一次对诊断上下文的迭代更新。通过不断地把新知识融入提示，系统能够在后续的诊断过程中更快地聚焦于高价值的检查路径，从而提升根因定位的效率和准确性。

通过这一机制，智能体不再是每次都从零开始推理，而是能够基于之前积累的经验不断优化自己的诊断上下文，从而逐步适应特定业务环境的故障模式与观测特征。这里所优化的上下文并非是以单次故障为目标的单次诊断情景，而是对整个目标微服务系统的长期适配过程：系统通过不断地把新知识融入提示，能够在后续的诊断过程中更快地聚焦于高价值的检查路径，从而提升根因定位的效率和准确性。

核心的离线演化工作流承担“主动学习”的职责，是 OpsEvolver 与传统被动告警系统最本质的区别。一次完整周期从 ChaosExplorer 开始，它会根据当前目标命名空间内的服务和 Pod 拓扑、历史注入记录以及候选动作集合，选择一个尚未覆盖的故障动作。故障注入完成后，AlertGenerator 会在预设观察窗口中采集指标、日志与链路数据，若检测到异常则生成统一的告警表示。随后，Diagnoser 围绕该告警执行探索式诊断，形成包含多个动作与观测结果的诊断轨迹；Introspector 再结合注入真值、告警信息和诊断轨迹，对诊断是否成功进行评估，并抽取知识条目与规则。周期结束时，知识条目会被写入知识库，注入记录、告警快照、轨迹摘要和生成规则则写入历史库，以支持后续追溯与重复注入规避。

离线演化中存在一个重要分支：如果故障注入后在观测窗口内没有触发任何异常告警，则该轮不会进入诊断与反思阶段，而是被记录为“无告警注入”。这类记录会在后续故障选择时回传给探索模块，用于降低重复探索无效区域的概率。这一机制既反映了环境的可观测性边界，也为后续实验分析中的注入有效率统计提供了直接依据。

这一流程的关键价值在于，它为智能体提供了一个无需人工标注即可积累经验的机制。即使某次诊断未能完全命中真实根因，系统也能从失败样例中识别出知识盲区；而即便某次注入未形成告警，系统也能据此更新探索边界，从而避免盲目重复。离线演化结束后，这些知识不会停留在描述层面，而会在后续基准评测中被重新用于诊断上下文，进一步检验其实际价值。

\section{多模态告警建模与可观测性设计}
\label{sec:alert_modeling_design}

OpsEvolver 并不把故障注入后的观测简化为单一信号，而是把指标、链路和日志作为三类互补证据统一建模。其核心判断依据不是“故障是否被注入”，而是“故障是否在既定工作负载与观测栈下产生了足够强的可学习症状”。这一设计直接决定了离线演化能否产出高价值样本。

\subsection{指标、链路与日志的互补关系}

三类遥测信号在语义上承担不同职责。指标适合描述资源占用、线程数、网络收发等连续数值型变化，能够较快识别资源饱和、流量异常和容器层面的状态漂移；链路追踪更适合刻画调用时延传播、上游等待与下游出错之间的因果传播路径，尤其适合分析网络退化、实例失效与调用链级联问题；日志则提供最直接的语义线索，能够暴露 \texttt{deadline exceeded}、\texttt{connection refused}、\texttt{panic} 等明确错误文本。

若只依赖其中一种信号，系统很容易在真实微服务环境中产生偏差。仅依赖指标时，可能只能看到某个上游服务的延迟升高，却无法判断问题来自下游依赖还是本地资源瓶颈；仅依赖日志时，容易被网关、中间层和 SDK 输出的表面错误误导；仅依赖链路时，则可能遗漏那些未明显拉长时延、但已在日志中清晰暴露的配置错误或连接错误。因此，OpsEvolver 从方法论上要求 AlertGenerator 同时覆盖三类信号，并在统一告警对象中保留证据来源。

\subsection{面向演化工作流的统一告警表示}

为了让诊断、反思和后续分析模块共享同一套输入，OpsEvolver 将指标、链路和日志中提取出的异常统一表示为一类标准化告警。每条告警至少包含名称、严重程度、关联服务、来源信号类型、核心数值证据以及自然语言描述。统一表示之后，诊断过程可以围绕同一语义接口规划动作，而不必针对不同信号源分别设计提示；反思过程也能够把“何种异常信号对应何类根因”抽象为知识条目与规则，而不受原始数据格式限制；历史记录和可视化分析则可以直接围绕统一告警表示开展统计，进一步分析故障到告警的映射关系、各系统的主导告警类型以及无告警区域分布。

\subsection{把无告警结果也视为系统知识}

在传统实验流程中，未触发告警的注入往往被简单视为无效运行；但对自主演化 SRE 而言，这类结果同样重要。因为它们揭示了给定环境中的真实边界：可能是工作负载没有覆盖到被干扰路径，也可能是观测窗口过短、阈值过保守，或者该类故障的业务外显程度本就较弱。

因此，OpsEvolver 在设计上要求告警生成机制明确区分“有告警”与“无告警”两类结果。前者进入诊断与反思，后者进入历史库中的负样本集合，用于约束后续故障探索。这使离线演化从单纯的样本采集，扩展为同时学习“有效区域”和“暂时不可观测区域”的边界学习过程。

\section{实验对象与工作负载设计}
\label{sec:benchmark_and_workload_design}

为了验证方法对不同业务拓扑和可观测特征的适配能力，OpsEvolver 选取了三个结构差异明显的微服务系统作为主要实验对象：Astronomy Shop、Hotel Reservation 和 Social Network。它们分别代表“链路丰富的电商系统”“以日志和错误率信号为主的预订系统”以及“调用链深、级联效应强的社交系统”。三者共同构成了一个较有代表性的云原生运维实验集合。

\subsection{Astronomy Shop：OpenTelemetry 电商示例系统}

Astronomy Shop 基于 OpenTelemetry Demo 构建，是一个典型的在线零售类微服务应用。系统包含前端代理、展示前端、购物车、结算、支付、配送、汇率换算、商品目录、推荐、报价、邮件通知和账务处理等多个业务服务，并天然集成了较完整的追踪埋点与负载生成组件。

该系统的特点是调用扇出明显、前后端路径丰富、链路观测完整，非常适合分析时延传播与依赖异常。例如，对支付或配送环节注入网络故障后，最先升高的往往不是故障服务自身的资源指标，而是结算或前端一侧的调用时延。也正因如此，Astronomy Shop 非常适合用于检验告警生成机制和诊断机制是否具备“由上游症状反推下游根因”的能力。

\subsection{Hotel Reservation：日志主导的预订系统}

Hotel Reservation 是一个围绕酒店查询、推荐、价格与预订流程构建的微服务系统，核心服务包括前端入口、检索、地理位置、价格计算、推荐、预订、用户和画像等，同时还依赖缓存、数据库和服务发现组件支撑完整业务流程。

与 Astronomy Shop 相比，该系统的链路结构更紧凑，但其错误日志更具辨识度。许多故障首先表现为前端请求失败、RPC 错误、服务发现异常或缓存/数据库访问问题，典型日志中常出现 \texttt{deadline exceeded}、\texttt{dial tcp}、\texttt{rpc error} 等文本。因此，该系统非常适合分析“日志信号强于资源信号”的场景，也能检验系统对配置失配、依赖超时和请求错误率上升的感知能力。

\subsection{Social Network：长调用链社交系统}

Social Network 选自 DeathStarBench 社交网络基准，包含入口网关、发帖编排、首页时间线、用户、社交关系、帖子存储、文本处理、短链接和媒体处理等大量细粒度服务。其核心写路径往往会跨越多个业务子系统，形成较长的依赖链。

这一系统最突出的特点是级联效应明显。许多下游故障并不会直接在故障服务本身形成最强告警，而是先在入口网关或中间层暴露为连接超时、写入失败或 Thrift 异常。对于面向根因定位的智能体而言，Social Network 是检验“症状在上游、根因在下游”这类复杂模式的重要实验对象。

\subsection{统一部署入口与工作负载驱动机制}

为了让故障演练建立在稳定且可重复的业务流量之上，OpsEvolver 在场景层为每个系统都封装了统一的部署、压测和清理机制。这样做可以保证每轮离线演化都运行在“系统已部署、流量已存在、故障可被放大”的统一前提下。

\subsection{Astronomy Shop 的内置负载生成}

Astronomy Shop 自带持续负载生成组件，能够模拟用户浏览、加购与下单等核心业务行为，因此无需叠加额外压测流量。这一设计使该系统的流量来源更接近其原始示例，也有利于保留 OpenTelemetry Demo 已预置的可观测性语义。

\subsection{Hotel Reservation 与 Social Network 的 CronJob 负载}

Hotel Reservation 与 Social Network 均通过 \texttt{wrk2}\cite{wrk2} 脚本生成外部请求，并由 Kubernetes CronJob 统一调度。CronJob 的调度表达式设置为每分钟触发一次；单次压测持续 50 秒，因此能够在相邻两次调度之间保留短暂空档，避免长时间重叠干扰，同时又能确保故障注入后的 5 分钟观测窗口内持续存在业务流量。

两套工作负载都采用指数分布到达过程与较小连接规模，以便在有限资源下稳定制造可观测请求。Hotel Reservation 主要对前端服务施加约 100 RPS 的读写混合流量；Social Network 则主要对发帖相关写路径施加约 10 RPS 的持续请求。两者均采用 2 个线程、2 个连接和 50 秒持续时间的配置，从而在维持场景可复现性的同时，为网络、Pod 和配置类故障提供足够的外显机会。

\subsection{工作负载设计对实验结论的影响}

可观测性并不是系统静态属性，而是故障、流量和监控三者共同作用的结果；没有稳定流量，再完备的告警生成器也无法产生有效样本。不同系统需要不同强度和不同业务路径的工作负载，才能把真实依赖问题放大为日志、指标或链路异常。此外，负样本分析之所以有价值，正是因为其背后绑定了具体的工作负载设置；当某一类故障频繁落入无告警集合时，后续优化方向可以是调整阈值，也可以是重新设计业务流量。

\subsection{基于 AIOpsLab 问题集的外部评测}

除长周期离线演化外，OpsEvolver 还引入了基于 AIOpsLab 问题集的独立基准评测，用于回答另一个同样关键的问题：系统通过演练积累下来的知识，是否真的能够改善根因定位。为此，OpsEvolver 从 Astronomy Shop、Hotel Reservation 和 Social Network 三个系统中选取全部可用问题，构成与前述实验对象一一对应的基准评测集合 \cite{chen2025aiopslab}。

这组基准评测的组织方式与离线演化不同。它不追求长期覆盖，而是围绕一组固定问题逐一运行相同的流程：先执行故障注入，再在统一观察窗口中生成告警，随后在同一组告警上分别运行两种诊断设置。第一种是不加入任何额外知识的基线诊断；第二种则把对应基底模型在长期演化中积累下来的知识条目与规则组织进上下文，形成知识增强诊断。通过这种并列对照，OpsEvolver 可以更直接地判断离线学习所得的知识是否真正改善了 RCA 表现。

\section{多智能体协作机制}
\label{sec:multi_agent_collaboration}

\subsection{角色划分与职责边界}

OpsEvolver 并未采用单一全能型 Agent，而是通过角色专门化降低复杂任务的认知负担。ChaosExplorer 聚焦于故障设计与执行，主要解决“向环境施加什么扰动”的问题；Diagnoser 聚焦于告警后的证据搜集与根因推理，主要解决“发生了什么、为什么发生”的问题；Introspector 聚焦于对诊断过程的元分析，主要解决“此次诊断有哪些经验可沉淀”的问题。三者在同一条离线演化链路上前后衔接，但各自面对的决策对象并不相同，因此也不需要共享完全一致的上下文。

其中，ChaosExplorer 与 Diagnoser 都可以看作符合 ReAct 范式的任务型智能体 \cite{yao2022react}。前者面对的是“理解当前系统，再决定施加何种故障”的问题，因此它既能执行故障注入动作，也能在注入前调用只读命令查询环境状态；后者面对的是“围绕告警继续搜集证据并收敛到根因”的问题，因此它会在诊断过程中交替进行推理、调用观测动作并根据新证据修正判断。与这两者不同，Introspector 不直接作用于运行环境，而是站在更高层面对一次完整演化周期进行评估和归纳，因此更接近一个以总结和抽象为主的分析角色。

在这种分工方式下，不同角色可采用不同的提示词、温度参数和动作约束。例如探索阶段允许更强的发散性，而知识增强诊断则强调更低温度和更少步骤。角色边界清晰后，系统更容易进行行为审计和结果解释，因为每一步动作都能被归因到特定模块。

\subsection{共享上下文与轨迹管理}

多智能体协作并不意味着任意共享长上下文。OpsEvolver 采用诊断轨迹与诊断报告作为核心中间表示：前者记录每一步的想法、动作、参数与观测结果，后者记录最终根因、置信度和证据链。对离线流程而言，反思模块通过读取完整轨迹了解诊断过程，并据此抽取知识与规则；在独立基准评测中，同一套轨迹与报告形式也被用来对比无知识诊断和知识增强诊断之间的差异。

这种“结构化状态共享”比直接共享原始对话历史更稳定，原因在于它减少了冗长语义上下文带来的噪声，并使历史经验具备机器可检索、可比较的特征。系统中的知识条目、周期记录和注入记录都建立在这种结构化共享思想之上。

\subsection{动作空间与执行约束}

三个角色虽然共享同一实验目标，但动作空间并不完全相同。ChaosExplorer 的动作空间由两部分组成：一部分是只读的环境查询动作，用于查看服务、Pod、部署和运行状态等信息；另一部分是受控故障动作，用于在既定边界内对目标系统施加扰动。Diagnoser 的动作空间则偏向观测，包括读取日志、导出指标、读取链路和执行短时命令等，其核心目的不是改变环境，而是围绕告警不断补充证据。Introspector 本身不再访问在线环境，而是读取注入真值、告警和诊断轨迹，从中判断诊断是否命中，并提炼可迁移的经验。

为了将 Agent 的行为限制在可审计范围内，OpsEvolver 把这些能力都整理为显式接口，并对执行边界进行约束。对于命令类动作，系统强调非交互、短时返回和以观测为主，避免把智能体放入不受控制的长期会话；对于故障注入动作，则要求其来自预先定义的受控动作集合，并具备明确的参数校验与恢复路径。这样一来，ChaosExplorer 和 Diagnoser 虽然都体现出 ReAct 式的“推理一动作一观察”特征，但它们的可执行能力始终处于清晰边界之内。

因此，OpsEvolver 的自治并不是“完全放开”，而是在一组声明式工具能力内进行有边界的自主推理。这一设计兼顾了实验灵活性与系统安全性。

\section{知识表示与自演化机制}
\label{sec:self_evolve_mechanism}

\subsection{知识条目与规则库}

OpsEvolver 将离线演化产生的经验整理成两类长期保存的知识。第一类是知识条目，面向具体症状模式，记录告警特征、推荐动作、适用条件、置信度以及所属演化周期；第二类是规则库，面向更通用的诊断经验，例如“网络异常应优先核查上下游依赖关系”或“高错误日志密度通常先于资源饱和出现”。前者更强调案例级复用，后者更强调跨案例迁移。

与直接微调模型相比，这种把经验保存在外部知识库中的方式更便于维护。知识可以按条目增删改查，不需要每次都重新训练模型；同时，知识内容与原模型参数解耦，也更适合变化较快的云环境。系统要修正已有经验时，只需更新知识库内容或调整规则优先级即可。

在OpsEvolver 实现中，知识条目与规则库承担了“连接学习与评测”的中介角色。离线阶段每完成一轮演化，系统都会把新的知识条目和规则增量写入持久化存储；在后续基准评测中，则根据同一问题触发的告警动态检索知识条目并装配规则提示。换言之，OpsEvolver 的学习结果并不隐藏在模型内部，而是以显式、可检查的形式不断生长。

除正向经验外，系统还通过历史库保存负向经验，即那些已经执行但未产生告警的故障组合。这部分信息虽然不会直接进入知识增强诊断的检索上下文，但可以影响离线阶段的故障探索，避免在无效区域重复探索浪费资源。因而，OpsEvolver 的“记忆”并不是单一知识库，而是由知识条目、规则库和负向探索边界共同构成的长期环境模型。

\subsection{知识增强诊断中的上下文组装}

在知识增强诊断中，系统需要从已有知识中挑选最有价值的上下文。当前系统采用轻量级关键词匹配方法对历史知识进行排序，其打分思想可以写为
\begin{equation}
    \label{eq:retrieval-score}
    score(k_i, a)=2 \cdot \mathbf{1}[n(a)\in p_i]+\mathbf{1}[s(a)\in p_i],
\end{equation}
其中，$a$ 表示当前告警，$k_i$ 表示第 $i$ 条知识条目，$n(a)$ 表示告警名称，$s(a)$ 表示告警严重程度，$p_i$ 表示知识条目的症状描述文本，$\mathbf{1}[\cdot]$ 为指示函数。若告警名称与知识条目症状匹配，则给予更高权重；严重等级匹配则给予辅助加分。

尽管这一检索方法相对简单，但它体现了OpsEvolver 的重要设计原则，即先利用已经积累的知识缩小诊断搜索空间，再让 LLM 在更小范围内继续推理。随着系统后续演进，该模块可以自然替换为向量检索或图检索，而不改变整体架构。

\section{系统设计特点与讨论}
\label{sec:design_discussion}

综合来看，OpsEvolver 的设计有几个比较鲜明的特点。系统以离线演化为组织核心，把故障注入、告警发现、证据搜集与知识提炼串联成一个端到端闭环，而不是若干彼此分离的模块；知识演化也始终围绕具体业务环境展开，强调针对特定系统逐步积累经验；同时，系统保留了较好的方法扩展性，无论是新增故障动作、替换检索算法，还是引入新的遥测源，都可以在不改变总体框架的前提下完成。

当然，该设计也存在权衡。这种把知识保存在外部的方式虽然灵活，但质量高度依赖知识抽取模块的表现；离线演练的覆盖程度也将直接决定后续基准评测中知识复用的上限；而无告警样本过多时，系统学到的更可能是边界而不是正向经验。上述问题将在第四章实验与分析中进一步讨论。
