\section{原型系统实现概览}
\label{sec:impl_overview}

OpsEvolver 原型采用 Python 实现，并围绕“工作流编排、智能体模块、遥测工具封装、知识持久化与场景部署”五类职责组织代码。系统入口类 \texttt{OpsEvolver} 负责实例化离线演化、在线诊断和反馈闭环三个工作流；各工作流进一步调用不同的 Agent、数据结构与工具接口完成具体任务。配置文件负责提供 Kubernetes 上下文、缓存目录、知识库存储路径以及大模型访问密钥等信息。

从实现角度看，本文原型并未追求复杂的分布式调度，而是优先保证流程完整性和接口清晰性。也就是说，系统首先要做到“从故障注入到知识沉淀的整条链路可以跑通”，再在此基础上逐步提升并发性、检索质量与评测规模。

\section{多维度故障注入系统}
\label{sec:fault_injection_impl}

\subsection{故障动作注册机制}

系统将故障注入能力统一抽象为 FaultAction 接口，所有具体故障动作都必须实现参数模式校验、执行逻辑与恢复逻辑。ChaosExplorer 在运行时只需要面对“动作名称+参数字典”这一统一接口，不需要关心底层是通过 Chaos Mesh、Kubernetes 资源修改还是宿主机 eBPF 程序完成注入。这种做法提升了系统的可扩展性，也使故障动作可以被安全地限制在一个白名单集合中。

\subsection{基于 Chaos Mesh 的声明式故障注入}

对于容器终止、Pod 失效以及网络扰动等常见云原生故障，本文优先采用 Chaos Mesh 的声明式资源对象实现。以 NetworkDelay 为例，系统根据智能体给出的目标微服务、延迟参数和抖动参数构造 NetworkChaos 对象，通过标签选择器将网络规则精准施加到目标服务对应的 Pod 上。由于 Chaos Mesh 原生支持实验删除与回收，这类故障具有较好的可回滚性。

相比直接执行宿主机级别的混沌脚本，声明式注入方式更符合 Kubernetes 环境下的工程实践。一方面，实验对象与参数都可显式记录，方便后续复盘；另一方面，借助统一的 CRD 形式，系统能够将不同类型的资源级故障纳入一致的编排逻辑中。

\subsection{部署失配与内核级故障注入}

仅依赖 Chaos Mesh 仍不足以覆盖真实运维中常见的配置错误与底层 I/O 异常。因此，本文进一步实现了两类补充性故障。

其一是 DeployMisconf 动作。该动作先通过 \texttt{kubectl get deployment -o yaml} 获取目标 Deployment 的完整描述，再使用 JSONPath 定位字段并修改其值，最后通过重新应用 YAML 的方式将错误配置写回集群。这使系统能够模拟诸如 CPU 限额配置不当、环境变量缺失、镜像参数偏差等高频配置问题。

其二是 DiskWoreOut 动作。该动作调用 eBPF 注入器定位指定进程，并针对特定系统调用返回错误码，从而模拟磁盘 I/O 故障。与容器级故障相比，这类问题往往更隐蔽，也更能检验智能体对底层异常与上层业务症状之间因果链的理解能力。当前原型通过受控二进制程序和进程名筛选来限制这类动作的影响范围。

\subsection{故障恢复与重复注入控制}

为了保证演练流程的可重复性和环境安全性，系统在每次成功注入故障后都会立即记录注入动作名称、参数和所属周期，并在周期结束后执行恢复逻辑。对于 Chaos Mesh 实验，恢复过程通常表现为删除对应实验资源；对于 Deployment 修改，则需要回写原始 YAML；对于 eBPF 故障，则需要清理内核挂载点或相关映射。

此外，HistoryBase 会持续维护历史注入记录。ChaosExplorer 在选择新故障时会检查历史记录，避免重复执行完全相同的故障动作与参数组合。这一机制能够提升离线演化的覆盖效率，使有限的实验预算更集中地探索未覆盖区域。

在当前版本中，这种去重逻辑已经从“只避免重复注入”扩展为“同时避免重复探索无告警区域”。HistoryBase 除了保存 \texttt{injected\_faults} 外，还会单独保存 \texttt{no\_alert\_faults}，并在新一轮注入决策时一并传入 ChaosExplorer。前者刻画“已经试过什么”，后者刻画“试过但当前不可观测的是什么”。这一扩展使离线演化更接近带约束的主动学习，而非简单随机注入。

\section{诊断智能体与动作执行}
\label{sec:diagnoser_impl}

\subsection{Plan-Act-Observe-Reflect 诊断循环}

Diagnoser 是系统中最核心的推理执行模块。其内部遵循 Plan-Act-Observe-Reflect 循环：首先根据告警和已有上下文生成当前诊断计划；随后从允许动作集合中选择一个动作并附带参数；动作执行后系统将观测结果写回上下文；最后模型据此反思当前假设是否成立，并决定继续收集证据还是提交根因结论。

当前原型在离线探索和在线推断两个场景中复用了同一诊断引擎，但使用不同的温度和步数上限。探索式诊断侧重多样化线索搜集，因此温度更高；在线推断更强调稳定收敛，因此温度更低、上下文更精简。

\subsection{动作接口设计}

为使诊断过程可控且易于审计，系统实现了如下几类标准动作接口。
\begin{enumerate}
    \item \textbf{日志动作。} 通过 \texttt{get\_logs} 读取指定命名空间和服务相关 Pod 的日志，并使用日志去重压缩算法减少重复探针输出对上下文的干扰。
    \item \textbf{命令动作。} 通过 \texttt{exec\_shell} 执行非交互式 shell 命令。系统会对 \texttt{kubectl edit}、持续跟随日志等高风险或不适合自动化的交互式命令进行拦截。
    \item \textbf{指标动作。} 通过 \texttt{get\_metrics} 导出 Prometheus 指标到本地目录，再由 \texttt{read\_metrics} 读取具体文件内容供模型分析。
    \item \textbf{链路动作。} 通过 \texttt{get\_traces} 从 Jaeger 提取追踪数据，再由 \texttt{read\_traces} 读取 CSV 形式的结构化结果。
\end{enumerate}

动作设计遵循“先采集、后阅读”的原则，即将大体量遥测数据先导出为文件，再按需读取具体片段，避免把大量原始数据直接压入模型上下文。这一设计对控制上下文长度和降低推理成本具有重要意义。

\subsection{探索式与推断式两种模式}

ExplorationDiagnoser 用于离线演化场景。代码中预设了若干不同风格的人格提示，例如偏应用层、偏基础设施、偏网络和偏数据路径等，以鼓励模型从不同视角观察同一告警。出于原型执行效率考虑，当前默认仅启用 \texttt{aggressive} 一种人格执行，但人格字典、批量返回结构和统一评估接口均已保留，为后续多路径并行诊断与投票式选择预留了实现空间。

InferenceDiagnoser 用于在线场景。它在诊断前先调用 ContextAssembler 拼装上下文，从知识库中检索相似案例，并从规则库中筛选可用规则，然后以低温度方式执行收敛式推断。也就是说，离线模式强调“多看、多试、多积累”，在线模式强调“少走弯路、快速落点”。

\subsection{诊断结果评估与成功判定}

相较于论文初稿阶段的概念性描述，当前原型在离线诊断后新增了明确的评估步骤。ExplorationDiagnoser 在获得不同人格的诊断轨迹后，会结合已知故障注入真值再发起一次独立评估，判断某条轨迹的最终根因是否“真正命中”了注入故障。若评估所依赖的大模型调用失败，系统还会回退到基于故障类型关键词和目标服务匹配的启发式判定。最终结果以 \texttt{evaluation\_success} 和 \texttt{evaluation\_reason} 的形式写入历史库，并作为后续图表分析和实验统计的依据。

这里需要区分两个层次的“成功”。第一层是 \texttt{Finished}，表示 Diagnoser 已经在最大步数限制内形成了提交结论；第二层是 \texttt{evaluation\_success}，表示该结论在语义上与真实故障类型或目标服务一致。两者并不等价：一个轨迹可以完成提交，但仍可能因为根因描述模糊、指向错误依赖或仅给出表面症状而在严格评估下被判为失败。该设计使实验分析不再只看“是否给出答案”，而是进一步看“答案是否真正对题”。

\subsection{置信度与证据链提取}

当前版本的 Diagnoser 在读取模型最终提交结果时，不再只抽取根因文本，而是同时解析 \texttt{Confidence:} 与 \texttt{Evidence:} 字段，并将其写入 DiagnosisReport。随后，系统还会把诊断步骤中的关键观测摘要追加到证据链中，形成“模型自述证据+执行轨迹证据”的组合式报告。这样做有两个好处：一方面，系统能够显式区分高置信度和低置信度结论，为后续反馈环节提供依据；另一方面，证据链可以与诊断步骤中的观测结果共同组成更完整的可解释诊断报告。

在实验统计中，\texttt{report\_confidence} 也被保留下来，成为分析不同系统诊断收敛性和知识成熟度的重要信号。

\section{遥测采集与告警生成}
\label{sec:telemetry_impl}

\subsection{基于指标的异常检测}

AlertGenerator 对指标异常的检测采用简单而稳健的 $3\sigma$ 思路。对于每一项容器级监控指标，系统同时计算短窗口均值与长窗口统计量，并据此估计标准化偏离程度。其核心思想可表示为
\begin{equation}
    \label{eq:zscore-metric}
    z_m(t)=\frac{\mu_{30s}(m,t)-\mu_{1d}(m,t)}{\sigma_{1d}(m,t)},
\end{equation}
其中，$\mu_{30s}(m,t)$ 表示指标 $m$ 在当前时刻附近短窗口内的均值，$\mu_{1d}(m,t)$ 和 $\sigma_{1d}(m,t)$ 分别表示该指标在较长历史窗口中的均值和标准差。当 $z_m(t)$ 超过阈值时，系统认为该指标出现异常漂移，并生成告警。

结合当前实现，系统优先关注 CPU、内存、线程与网络等容器级常见指标，如 \texttt{container\_cpu\_usage\_seconds\_total}、\texttt{container\_memory\_usage\_bytes} 与若干网络收发统计量。这种设计在原型阶段的优点是实现成本低、适用面广，缺点是尚未引入更复杂的多指标关联检测。

\subsection{基于链路与日志的异常检测}

除指标外，系统还从 Jaeger 和日志接口中提取告警信号。链路检测模块负责抓取各服务跨度信息，并把服务名、操作名、时延、父子关系和错误标记等字段整理为 DataFrame，以便后续分析。当前原型已经打通链路抓取、结构化转换与文件导出链路，并预留了基于 P95 延迟阈值触发异常的逻辑接口。

日志检测模块则采用关键词计数策略。在固定时间窗口内，系统统计每个 Pod 日志中 \texttt{ERROR}、\texttt{FAIL} 和 \texttt{EXCEPTION} 等错误关键词的出现频次；当计数超过预设阈值时，生成严重级别的日志异常告警。尽管这种方法较为朴素，但对于原型验证而言足以捕获大量由配置错误、依赖异常和启动失败导致的显性故障。

\subsection{统一告警对象建模}

无论告警来自指标、链路还是日志，系统都会将其统一表示为 Alert 对象。该对象至少包含告警名称、严重程度、影响服务、数值型证据和自然语言描述等字段。统一建模的意义在于：后续诊断智能体无需关心告警来源，只需要围绕这一统一接口组织排障路径；与此同时，知识库中的症状模式也能够围绕 Alert 语义进行检索与匹配。

从当前实验数据来看，不同系统的主导告警类型差异明显：Astronomy Shop 更容易触发 \texttt{HighLatency}，而 Hotel Reservation 和 Social Network 更频繁地表现为 \texttt{ExcessiveErrorsInLogs}。统一告警对象的存在，使这些跨系统差异能够在同一分析框架下被比较。

\section{知识库与历史库持久化}
\label{sec:knowledge_base_impl}

\subsection{知识条目数据结构}

本文使用 KnowledgeItem 数据结构表示单条可复用运维经验。每条知识至少包含如下字段：唯一标识、症状模式、推荐动作、适用条件、置信度、来源、描述信息、演化周期编号与时间戳。其中，症状模式描述“看到了什么”，推荐动作描述“接下来该做什么”，适用条件描述“在什么前提下这条经验有效”，演化周期编号则保证知识来源可追溯。

这一设计与传统 FAQ 或纯文本排障手册不同。KnowledgeItem 是面向机器检索与上下文组装设计的结构化单元，既能被诊断时引用，也能在后续反馈中被修正或降权。

\subsection{知识库存储与更新}

KnowledgeBase 将知识条目和规则库统一保存在 JSON 文件中。离线工作流每完成一轮演化，系统就将新产生的知识条目追加到内存集合，并立即持久化到磁盘。当前原型未直接集成向量数据库，而是采用文件级持久化与轻量级关键词匹配的方式满足原型验证需求。此种方案虽然在大规模场景下检索能力有限，但胜在实现简单、可读性强、便于人工审查。

\subsection{历史记录与可追溯性}

除知识库外，系统还专门设计了 HistoryBase 记录每一轮演化的完整上下文，包括注入事件、告警快照、诊断轨迹摘要、生成的知识条目标识以及规则集合。当前版本还在 HistoryBase 中单独维护了 \texttt{no\_alert\_injections} 字段，用于记录“成功执行但未产生任何可观测告警”的故障注入。

与只保存最终根因不同，这种历史记录方式使研究者能够回答如下问题：某条知识是在哪一轮演化中生成的；哪种故障注入最容易产生高价值知识；某次失败诊断究竟败在检索、规则还是观测数据不足；哪些故障在当前工作负载和观测窗口下难以形成有效样本。第四章中的有效注入率、无告警分布以及故障-告警关联分析，均建立在这些持久化记录之上。

\section{实验分析数据管线}
\label{sec:analysis_pipeline}

为了将原始历史记录转化为论文中的统计结果，当前版本额外实现了一套独立于核心工作流的分析与可视化模块，位于 \texttt{opsevolver/visualize/} 目录下。其核心思路是将 HistoryBase 和 KnowledgeBase 中的持久化 JSON 快照扁平化为逐周期样本表，再围绕这些样本表生成统计图和分析结果。

\subsection{逐周期样本表构造}

\texttt{visualize/\_data.py} 中的 \texttt{load\_raw} 会直接读取各命名空间下最新的 \texttt{v1} 历史快照与知识快照，而 \texttt{build\_cycle\_df} 会把每一轮有效演化周期整理为一行结构化样本，主要字段包括：命名空间、故障类型、目标服务列表、告警数量、告警类型集合、诊断结果、报告置信度、严格评估成功标记、诊断步数、该周期生成的知识条目数以及规则数。这种“每轮演化一条样本”的数据表示方式，使离线演化实验能够被当作一个随时间演进的数据集来分析。

\subsection{统计图与分析维度}

基于上述样本表，系统进一步实现了多类统计图脚本，包括故障分布、诊断结果、知识时间线、故障与服务覆盖热力图、故障到告警的关系图、知识库深度分析以及无告警注入分析等。这些脚本并不改变核心诊断逻辑，而是为论文实验部分提供了一套可复用的分析视角。特别是无告警注入分析脚本直接读取 HistoryBase 中的 \texttt{no\_alert\_injections} 字段，从而把过去在实验中常被忽略的“负样本”也纳入统一统计口径。

从研究方法上看，这意味着 OpsEvolver 不仅能“运行实验”，还具备将实验结果结构化、可视化和可对比化的能力。对于后续扩展更大规模实验、做版本间对比或加入新的被测系统，这一点尤为重要。

因此，HistoryBase 是支撑系统复盘、去重和未来量化评估的重要基础设施。

\section{实验场景部署与工作负载驱动}
\label{sec:scenario_impl}

为验证框架在不同业务特征下的适用性，原型接入了多个典型微服务应用场景，包括 Hotel Reservation、Social Network、Astronomy Shop 和 Train Ticket。不同应用由独立的 Application 子类封装部署与清理逻辑，既可以通过 Helm Chart 部署，也可以通过 Kubernetes YAML 直接下发。

在工作负载驱动方面，Hotel Reservation 与 Social Network 通过 \texttt{wrk} 脚本向前端服务持续发压，分别模拟酒店推荐与社交动态发布等核心路径；Astronomy Shop 使用系统内置负载生成器；Train Ticket 当前主要保留为大规模微服务部署对象，用于验证框架对复杂拓扑的适配能力。场景层的封装使论文后续实验可以基于统一接口切换被测对象，而不必修改核心诊断流程。

\section{本章小结}
\label{sec:chap03_summary}

本章从故障注入、诊断引擎、遥测采集、告警生成、知识持久化和场景部署六个方面介绍了 OpsEvolver 的具体实现。可以看到，系统并不是单纯围绕 LLM 组织，而是将大模型能力嵌入到故障演练、环境交互和知识管理的整体工程流程中。基于这些实现，下一章将进一步分析该原型在实验对象、能力覆盖和典型案例上的表现。
