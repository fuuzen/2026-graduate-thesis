\section{实验目的与研究问题}
\label{sec:exp_goal}

在完成原型系统实现之后，OpsEvolver 进一步希望回答两个更具体的问题：OpsEvolver 是否能够在真实微服务环境中持续积累可学习的运维样本，并把这些样本转化为可复用的知识与规则；而这些知识在独立的 RCA 任务中又是否真的有助于提升诊断效果。为此，本章不再停留于单次演示，而是同时基于三个典型微服务系统上的长周期离线演化数据和基于 AIOpsLab 问题集的基准评测结果，对系统的样本规模、可观测性边界、告警模式、诊断行为、知识增长过程以及知识复用效果进行系统分析。

为使本章分析主线更清晰，OpsEvolver 将关注的问题整理为\autoref{tab:research-questions} 所示的六个研究问题。

\section{实验设置与数据来源}
\label{sec:exp_setup}

\subsection{实验软件环境}

实验运行环境建立在 Kubernetes 集群之上，并借助多种可观测与混沌工程组件支撑完整闭环。主要软件栈包括：用于部署和命名空间隔离的 Kubernetes，用于声明式故障注入的 Chaos Mesh \cite{chaosmesh}，用于指标采集的 Prometheus \cite{prometheus}，用于调用链观测的 Jaeger \cite{jaeger}，以及作为智能体底座模型的 DeepSeek-V3.2 \cite{liu2025deepseek}、Llama 4 Maverick 17B Instruct \cite{adcock2026llama} 和 Qwen3-32B \cite{yang2025qwen3}。基于这些组件，OpsEvolver 能够在统一工作流中串联故障注入、告警生成、诊断推理和知识沉淀。

\subsection{被测系统与实验口径}

OpsEvolver 定量分析聚焦于三个已形成较完整实验数据的系统：Astronomy Shop、Hotel Reservation 和 Social Network。三者在第二章中已经介绍了其业务特征与工作负载设计，这里不再赘述。本章实验分为两层。第一层围绕 DeepSeek-V3.2 对应的主实验历史记录，分析系统行为、可观测性边界和知识增长过程；第二层则引入另外两种基底模型的历史记录，比较不同模型对离线演化路径的影响。需要特别说明的是，模型对比部分为了保证公平性，对每个模型在每个系统上的实验都只截取前 300 轮注入记录，因此三种模型的对比都建立在统一的“每系统 300 轮、每模型 900 轮”口径上。

因此，主实验中的 DeepSeek-V3.2 仍使用其完整历史数据描述系统总体行为，而模型对比部分则不再直接使用各模型的全量注入总数做优劣判断，而是统一截取前 300 轮记录，并围绕可观测有效率、诊断完成率、严格命中率以及单位有效周期的知识沉淀密度等指标展开比较。

\subsection{独立基准测评的评测口径}

除上述长周期演化数据外，OpsEvolver 还使用 AIOpsLab 问题集构造了独立 benchmark，用于比较无知识基线诊断与知识增强诊断 \cite{chen2025aiopslab}。在这一评测中，三个系统分别对应 11、11 和 6 个真实故障问题，并各自包含 1 个无故障样本。对于每个问题，系统都先完成故障注入与告警生成，再在同一组告警上分别运行两种诊断设置。由于基准测评的目的在于观察知识复用效果，因此本节主要报告能够成功进入诊断阶段的问题；无故障样本仅作为误报观察的辅助信息，不纳入主表中的 RCA 正确率分母。

基准测评实验使用了两种基底模型 DeepSeek-V3.2 与 Qwen3-32B。两组知识增强诊断分别调用各自在长期演化中沉淀下来的知识与规则，因此可以直接观察“同一基底模型在有无历史知识条件下”的差异。

\subsection{核心评价指标}

为了避免不同统计对象之间混淆，OpsEvolver 采用如下三项核心指标。
\begin{equation}
    \label{eq:obs-rate}
    \eta_{obs} = \frac{N_{eff}}{N_{inj}},
\end{equation}
其中，$N_{inj}$ 表示总故障注入次数，$N_{eff}$ 表示成功触发至少一条告警并进入诊断流程的有效周期数，$\eta_{obs}$ 表示注入可观测有效率。

\begin{equation}
    \label{eq:finish-rate}
    \rho_{fin} = \frac{N_{fin}}{N_{eff}},
\end{equation}
其中，$N_{fin}$ 表示在限定步数内形成完整诊断结论的周期数，$\rho_{fin}$ 表示诊断完成率。

\begin{equation}
    \label{eq:hit-rate}
    \rho_{hit} = \frac{N_{hit}}{N_{eff}},
\end{equation}
其中，$N_{hit}$ 表示经严格评估后被判定为命中真实故障的周期数，$\rho_{hit}$ 表示严格根因命中率。

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.65\textwidth]{image/chap04/dpsk/no_alert_analysis.png}
    \caption{无告警样本分析}

从系统层面看，Astronomy Shop 没有出现无告警样本，说明其内置负载、丰富埋点和调用扇出结构能够稳定放大故障影响；Hotel Reservation 与 Social Network 则分别存在 143 和 113 个无告警样本，表明日志、指标和链路虽然能够捕获大量异常，但仍不能覆盖所有扰动组合。

从故障类型看，Hotel Reservation 中无告警样本主要集中在 \texttt{NetworkDelay}（48 次）、\texttt{PodKill}（25 次）和 \texttt{PodFailure}（23 次）；Social Network 中则以 \texttt{NetworkDelay}（36 次）、\texttt{PodKill}（30 次）和 \texttt{NetworkLoss}（14 次）为主。该结果说明，即便是最常见的基础设施故障，如果没有足够强的业务流量放大或没有命中关键调用路径，也可能无法转化为对智能体有学习价值的样本。

\section{告警模式与服务覆盖}
\label{sec:alert_patterns}

\subsection{故障到告警的映射关系}

\autoref{fig:dpsk-fault-alert-graph} 给出了三个系统中故障类型到告警类型的关系图。该图的价值在于，它不只回答“什么故障发生得多”，还回答“什么故障最终被告警生成机制感知为哪类症状”。

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.95\textwidth]{image/chap04/dpsk/fault_alert_graph.png}
    \caption{故障类型到告警类型的关系图}
    \label{fig:dpsk-fault-alert-graph}
\end{figure}

从图中可以提炼出清晰的系统差异。Astronomy Shop 中最粗的边主要连接到 HighLatency 和 ExcessiveErrorsInLogs，尤其是 NetworkDelay$\rightarrow$HighLatency（79 次）、NetworkDelay$\rightarrow$ExcessiveErrorsInLogs（72 次）以及 PodKill$\rightarrow$HighLatency（53 次）。这说明在该系统中，网络和实例故障很容易沿调用链放大为显著的延迟异常。

Hotel Reservation 中的主导边则集中在日志和错误率方向，最常见模式是 NetworkDelay$\rightarrow$ExcessiveErrorsInLogs（37 次）、PodKill$\rightarrow$ExcessiveErrorsInLogs（27 次）以及 NetworkDelay$\rightarrow$HighErrorRate（15 次）。这说明相较于链路时延，日志与请求错误率更能稳定反映该系统中的依赖问题。

Social Network 的关系图则呈现更明显的“日志主导”特征，其中 NetworkDelay$\rightarrow$ExcessiveErrorsInLogs 达到 75 次，PodFailure$\rightarrow$ExcessiveErrorsInLogs 达到 67 次，远高于 HighLatency 相关边。这与其网关、中间层和深调用链结构高度一致，即许多下游故障会优先在上游连接日志中暴露出来。

\subsection{不同系统的主导告警模式}

结合历史记录进一步统计可知，Astronomy Shop 共产生 531 条告警，其中 \texttt{HighLatency} 277 条、\texttt{ExcessiveErrorsInLogs} 254 条；Hotel Reservation 共产生 180 条告警，其中 \texttt{ExcessiveErrorsInLogs} 131 条、\texttt{HighErrorRate} 49 条；Social Network 共产生 260 条告警，其中 \texttt{ExcessiveErrorsInLogs} 232 条，\texttt{HighLatency} 仅 26 条，\texttt{HighErrorRate} 只有 2 条。

这说明统一的告警生成机制在不同场景中学到的“主导观测入口”并不相同。Astronomy Shop 更接近追踪和时延主导的系统，适合学习“高延迟在上游出现、根因在下游”的模式；Hotel Reservation 更接近“日志+错误率”主导的系统；Social Network 则更强调“通过上游日志反推复杂下游依赖”的排障路径。

\section{AIOpsLab 基准测评上的知识增强诊断对比}
\label{sec:benchmark_comparison}

长周期演化实验回答的是“系统能否学到知识”，而独立基准测评更关心“这些知识是否真的有助于 RCA”。为此，OpsEvolver 把三个系统中的 AIOpsLab 问题逐一运行，并在相同告警输入上对比无知识基线诊断和知识增强诊断。

\subsection{结果分析}

更细致的结果如\autoref{tab:benchmark-by-system} 所示，知识的帮助对于两种模型并不一致。Astronomy Shop 中，DeepSeek-V3.2 的正确率从 0 提升到 27.3\%，而 Qwen3-32B 则保持在 18.2\% 不变。这表明在 OpenTelemetry 电商示例中，知识增强更像是在为 DeepSeek-V3.2 补足环境经验，而对 Qwen3-32B 的边际作用较小。

Hotel Reservation 上的结果更能体现知识复用的直接收益。DeepSeek-V3.2 正确率从 0 提升到 27.3\%；Qwen3-32B 正确率则从 63.6\% 提升到 81.8\%。考虑到该系统中的问题大多与数据库认证、依赖服务异常和错误率升高相关，这一结果说明，长期演化中沉淀下来的日志模式、依赖关系和规则提示，确实能够帮助模型更快收束到真正的故障点。

\section{典型案例分析}
\label{sec:case_study}

为了避免案例分析停留在概念层面，本节直接结合主实验历史记录中的具体周期，对三套系统各选取一组具有代表性的故障注入与诊断样本进行讨论。所选样本均来自 DeepSeek-V3.2 对应的离线演化历史，能够同时反映故障动作、告警表现、诊断结果以及知识沉淀之间的关系。

\subsection{Astronomy Shop：时延传播主导的电商链路}

Astronomy Shop 中最有代表性的现象，是故障症状常常首先出现在调用链的上游，而不是直接停留在被注入组件自身。以一次对结算服务施加了 200\,s 的 PodKill 故障为例，告警并没有首先表现为结算服务自身崩溃，而是以前端服务的 \texttt{HighLatency} 形式出现，其 P95 时延达到 23925\,ms，P99 达到 63898\,ms；与此同时，还伴随一条来自评论服务的日志异常告警。这一周期中的诊断轨迹共执行了 3 步，但最终仍返回 \texttt{Unknown}，未能通过严格评估，也没有沉淀出新的知识条目。这个样本很典型地说明，在电商链路中，根因位于下游结算环节时，最先放大的症状可能却落在前端与入口路径上，而并发日志噪声又会进一步增加定位难度。

与之相对的另一轮迭代中，系统对账务处理服务施加了 80\,s 的 PodFailure 故障，随后在链路侧捕获到账务处理服务 P95 时延升高到 38431\,ms，主要异常操作集中在订单接收与订单消费路径；同时还出现了一条较弱的评论服务日志异常。与前一个失败案例不同，这一次诊断结果明确指出账务处理服务存在严重时延问题，并顺利通过严格评估，随后生成了 1 条知识和 1 条规则。结合这两个案例可以发现，Astronomy Shop 的价值并不只是“容易产生高时延”，更在于它持续迫使系统学习一种更符合微服务实际的诊断模式，即高时延告警可能首先出现在上游入口，而根因需要沿调用链向下游逆向追踪。

\subsection{Hotel Reservation：日志与错误率优先暴露依赖问题}

Hotel Reservation 的代表性现象是日志与错误率往往比资源或时延指标更早暴露根因。一个很典型的案例是：系统对画像数据库容器（hotel-reserv-profile-mongo）执行了容器终止故障。随后，链路侧首先在前端入口检测到 HighErrorRate 告警，错误率达到 23.8\%（2410 个跨度中有 574 个失败），但其 P95 时延只有 6\,ms；紧接着，日志侧又在前端 Pod 中捕获到 6002 条失败日志，代表性文本为 SearchHandler GetProfiles failed。这组信号非常有代表性，因为它表明系统并没有被“低时延”所误导，而是从高错误率和下游调用失败日志中识别出问题。

该周期的诊断结果最终将根因定位为画像服务不可用，指出前端对获取用户画像的 gRPC 调用因下游服务缺失而失败，严格评估判定为命中，并由此沉淀出“当上游日志明确指向下游调用失败时，应优先检查下游服务可用性与容器状态”的规则。这类样本说明，Hotel Reservation 中许多故障虽然最终表现为前端失败，但最有效的证据往往不是资源侧异常，而是错误率与日志文本中对下游依赖的直接指向。因此，该系统对 OpsEvolver 的主要价值，在于帮助其积累“从日志与 RPC 错误中快速识别依赖失效”的知识模式。

\subsection{Social Network：由上游日志反推深层下游根因}

Social Network 最具代表性的价值在于级联故障学习，即最醒目的症状先出现在入口或中间层，而真正的性能瓶颈位于更深的下游链路。例如，某一次系统对媒体前端、媒体处理、短链接、文本处理和用户提及等多个服务同时注入了 180\,s 的网络时延，延迟幅度达到 35\,s，并带有 800\,ms 抖动。故障注入后，首先被捕获到的是入口 Web 服务的 \texttt{HighLatency} 告警，其 P95 时延达到 40930\,ms，错误率达到 17.2\%；与此同时，入口网关日志中出现了 804 条错误，既包括 \texttt{tcp socket read timed out}，也包括 \texttt{TTransportException} 和连接被拒绝等异常。

如果只看入口侧现象，很容易把问题简单归结为网关或发帖编排本身过载；但历史记录中的链路摘要显示，真正被明显拉长的是发帖编排服务，其 P95 时延达到 10519\,ms，而文本处理服务和用户提及服务的 P95 时延也上升到约 300\,ms。诊断结果最终指出，入口网关在调用发帖编排服务时发生超时，而更深层的文本处理与用户提及链路变慢进一步放大了这一问题，严格评估将其判定为成功。该周期随后生成了“当入口同时出现高时延和大量超时日志时，应优先检查服务间网络连通性与下游时延传播”的规则。这个案例说明，Social Network 中最有价值的不是单个告警本身，而是系统已经开始学会把入口症状、中间层超时和下游服务退化串联起来，逐步形成“症状在上游、根因在下游”的级联诊断经验。

\section{局限性分析}
\label{sec:limitations}

尽管实验数据已经显著丰富，但结果也清楚暴露出系统边界。首先，总体严格命中率只有 3.4\%，说明系统距离稳定而精确的根因定位仍有明显差距。其次，Hotel Reservation 与 Social Network 的无告警样本分别达到 143 个和 113 个，其原因可能是工作负载与阈值策略还没有把所有故障都充分放大为可学习信号。

此外，知识条目和规则虽然持续增长，但知识增强诊断目前仍主要依赖关键词匹配与规则拼装，图结构信息和语义检索的利用还比较有限；现有实验也主要围绕单条主诊断轨迹展开，尚未系统评估多人格并行探索与结果投票的收益。最后，本章主要覆盖单类故障演练，距离真实生产中多故障叠加、跨系统联动的复杂场景仍有不小差距。

\section{本章小结}
\label{sec:chap04_summary}

本章基于 DeepSeek-V3.2 主实验数据、前三百轮基底模型消融实验以及 AIOpsLab 基准测评对比实验，对 OpsEvolver 在 Astronomy Shop、Hotel Reservation 和 Social Network 三个系统上的表现进行了数据驱动分析。其中 Astronomy Shop 的可观测性最强，而 Hotel Reservation 与 Social Network 中仍存在显著无告警区域；在每个系统统一截取前 300 轮之后，不同基底模型体现出“可观测覆盖、严格命中与知识沉淀密度”之间的明显权衡；而在独立基准测评上，知识增强诊断又进一步证明，离线演化得到的知识在 DeepSeek-V3.2 与 Qwen3-32B 上都能带来不同程度的 RCA 收益。总体而言，OpsEvolver 已经具备了持续积累、持续分析并暴露自身边界的研究平台特征。
