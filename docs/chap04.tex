\chapter{实验与分析}
\label{cha:experiments}

\section{实验目的与研究问题}
\label{sec:exp_goal}

在完成原型系统实现之后，本文进一步希望回答一个更具体的问题：OpsEvolver 是否能够在真实微服务环境中持续积累可学习的运维样本，并把这些样本转化为可复用的知识与规则。为此，本章不再停留于单次演示，而是基于三个典型微服务系统上的长周期离线演化数据，对系统的样本规模、可观测性边界、告警模式、诊断行为和知识增长过程进行系统分析。

本文围绕以下四个研究问题展开。
\begin{enumerate}
    \item \textbf{RQ1：} 以 DeepSeek-V3.2 主实验为例，OpsEvolver 在三个微服务系统上累计形成了多大规模的实验资产？
    \item \textbf{RQ2：} 不同系统、不同故障类型的可观测性有何差异，哪些故障更容易落入无告警区域？
    \item \textbf{RQ3：} 在统一的 AlertGenerator 机制下，不同系统会呈现怎样的故障-告警耦合关系和服务覆盖特征？
    \item \textbf{RQ4：} 系统在诊断完成、严格根因命中以及知识沉淀方面表现如何，主要瓶颈位于哪个阶段？
    \item \textbf{RQ5：} 当基底模型替换为 Llama 4 Maverick 17B Instruct 或 Qwen3-32B 时，系统的可观测覆盖、诊断行为与知识沉淀会发生怎样的变化？
\end{enumerate}

\section{实验设置与数据来源}
\label{sec:exp_setup}

\subsection{实验软件环境}

实验运行环境建立在 Kubernetes 集群之上，并借助多种可观测与混沌工程组件支撑完整闭环。主要软件栈包括：用于部署和命名空间隔离的 Kubernetes，用于声明式故障注入的 Chaos Mesh \cite{chaosmesh2022}，用于指标采集的 Prometheus \cite{prometheus2024}，用于调用链观测的 Jaeger \cite{jaeger2024}，以及作为智能体底座模型的 DeepSeek-V3.2 \cite{liu2025deepseek}、Llama 4 Maverick 17B Instruct \cite{adcock2026llama} 和 Qwen3-32B \cite{yang2025qwen3}。基于这些组件，OpsEvolver 能够在统一工作流中串联故障注入、告警生成、诊断推理和知识沉淀。

\subsection{被测系统与实验口径}

本文定量分析聚焦于三个已形成较完整实验数据的系统：Astronomy Shop、Hotel Reservation 和 Social Network。三者在第二章中已经介绍了其业务特征与工作负载设计，这里不再赘述。本章实验分为两层。第一层以 \texttt{dpsk.json} 为主实验数据，围绕系统行为、可观测性边界和知识增长进行细致分析；第二层引入 \texttt{llama.json} 和 \texttt{qwen.json} 作为消融实验数据，比较不同基底模型对离线演化路径的影响。

需要说明的是，三组模型实验运行轮次并不完全一致，因此绝对注入总数主要用于描述实验资产规模，而不同模型之间的优劣比较更依赖可观测有效率、诊断完成率、严格命中率以及单位有效周期的知识沉淀密度等归一化指标。

\subsection{核心评价指标}

为了避免不同统计对象之间混淆，本文采用如下三项核心指标。
\begin{equation}
    \label{eq:obs-rate}
    \eta_{obs} = \frac{N_{eff}}{N_{inj}},
\end{equation}
其中，$N_{inj}$ 表示总故障注入次数，$N_{eff}$ 表示成功触发至少一条告警并进入诊断流程的有效周期数，$\eta_{obs}$ 表示注入可观测有效率。

\begin{equation}
    \label{eq:finish-rate}
    \rho_{fin} = \frac{N_{fin}}{N_{eff}},
\end{equation}
其中，$N_{fin}$ 表示诊断结果状态为 \texttt{Finished} 的周期数，$\rho_{fin}$ 表示诊断完成率。

\begin{equation}
    \label{eq:hit-rate}
    \rho_{hit} = \frac{N_{hit}}{N_{eff}},
\end{equation}
其中，$N_{hit}$ 表示通过 \texttt{evaluation\_success} 严格判定为命中真实故障的周期数，$\rho_{hit}$ 表示严格根因命中率。

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.95\textwidth]{image/chap04/dpsk/no_alert_analysis.png}
    \caption{无告警样本分析}
    \label{fig:dpsk-no-alert}

从系统层面看，Astronomy Shop 没有出现无告警样本，说明其内置负载、丰富埋点和调用扇出结构能够稳定放大故障影响；Hotel Reservation 与 Social Network 则分别存在 143 和 113 个无告警样本，表明日志、指标和链路虽然能够捕获大量异常，但仍不能覆盖所有扰动组合。

从故障类型看，Hotel Reservation 中无告警样本主要集中在 \texttt{NetworkDelay}（48 次）、\texttt{PodKill}（25 次）和 \texttt{PodFailure}（23 次）；Social Network 中则以 \texttt{NetworkDelay}（36 次）、\texttt{PodKill}（30 次）和 \texttt{NetworkLoss}（14 次）为主。该结果说明，即便是最常见的基础设施故障，如果没有足够强的业务流量放大或没有命中关键调用路径，也可能无法转化为对智能体有学习价值的样本。

\section{告警模式与服务覆盖}
\label{sec:alert_patterns}

\subsection{故障到告警的映射关系}

\autoref{fig:dpsk-fault-alert-graph} 给出了三个系统中故障类型到告警类型的关系图。该图的价值在于，它不只回答“什么故障发生得多”，还回答“什么故障最终被 AlertGenerator 感知为哪类症状”。

\begin{figure}[p]
    \centering
    \includegraphics[width=0.95\textwidth]{image/chap04/dpsk/fault_alert_graph.png}
    \caption{故障类型到告警类型的关系图}
    \label{fig:dpsk-fault-alert-graph}
\end{figure}

从图中可以提炼出清晰的系统差异。Astronomy Shop 中最粗的边主要连接到 \texttt{HighLatency} 和 \texttt{ExcessiveErrorsInLogs}，尤其是 \texttt{NetworkDelay$\rightarrow$HighLatency}（79 次）、\texttt{NetworkDelay$\rightarrow$ExcessiveErrorsInLogs}（72 次）以及 \texttt{PodKill$\rightarrow$HighLatency}（53 次）。这说明在该系统中，网络和实例故障很容易沿调用链放大为显著的延迟异常。

Hotel Reservation 中的主导边则集中在日志和错误率方向，最常见模式是 \texttt{NetworkDelay$\rightarrow$ExcessiveErrorsInLogs}（37 次）、\texttt{PodKill$\rightarrow$ExcessiveErrorsInLogs}（27 次）以及 \texttt{NetworkDelay$\rightarrow$HighErrorRate}（15 次）。这说明相较于链路时延，日志与请求错误率更能稳定反映该系统中的依赖问题。

Social Network 的关系图则呈现更明显的“日志主导”特征，其中 \texttt{NetworkDelay$\rightarrow$ExcessiveErrorsInLogs} 达到 75 次，\texttt{PodFailure$\rightarrow$ExcessiveErrorsInLogs} 达到 67 次，远高于 \texttt{HighLatency} 相关边。这与其网关、中间层和深调用链结构高度一致，即许多下游故障会优先在上游连接日志中暴露出来。

\subsection{不同系统的主导告警模式}

结合历史记录进一步统计可知，Astronomy Shop 共产生 531 条告警，其中 \texttt{HighLatency} 277 条、\texttt{ExcessiveErrorsInLogs} 254 条；Hotel Reservation 共产生 180 条告警，其中 \texttt{ExcessiveErrorsInLogs} 131 条、\texttt{HighErrorRate} 49 条；Social Network 共产生 260 条告警，其中 \texttt{ExcessiveErrorsInLogs} 232 条，\texttt{HighLatency} 仅 26 条，\texttt{HighErrorRate} 只有 2 条。

这说明统一的 AlertGenerator 在不同场景中学到的“主导观测入口”并不相同。Astronomy Shop 更接近追踪和时延主导的系统，适合学习“高延迟在上游出现、根因在下游”的模式；Hotel Reservation 更接近“日志+错误率”主导的系统；Social Network 则更强调“通过上游日志反推复杂下游依赖”的排障路径。

综合\autoref{tab:service-density-dpsk} 和两幅知识演化图可以发现，三套系统都能维持稳定的知识沉淀速度：Astronomy Shop 平均每个有效周期产生 1.052 条知识和 1.131 条规则，Hotel Reservation 分别为 0.994 和 1.051，Social Network 分别为 1.000 和 1.071。这意味着 OpsEvolver 的外部记忆正在以接近线性的方式随实验数据增长，而不是停留在少量代表性案例的人工整理阶段。

\subsection{主要瓶颈的重新定位}

结合\autoref{tab:dpsk-overview} 与\autoref{tab:diagnosis-quality-dpsk} 可以发现，系统瓶颈已经从单一问题演变为两个阶段性问题并存。
\begin{enumerate}
    \item \textbf{阶段一：可观测性不足。} Hotel Reservation 与 Social Network 中仍存在大量无告警样本，说明工作负载、观测窗口和阈值设置仍有优化空间。
    \item \textbf{阶段二：严格落点能力不足。} 即使已经进入有效周期，系统的严格根因命中率依然偏低，说明从症状到根因的最后一步映射仍然是最难部分。
\end{enumerate}

这一结论的意义在于，OpsEvolver 已经不是“完全跑不通”的原型，而是一个能够明确暴露自身短板的研究系统。它既知道自己在哪些区域看不见，也知道自己在哪些区域虽然看见了，但还没有真正说对。

\section{典型案例分析}
\label{sec:case_study}

\subsection{Astronomy Shop：时延传播主导的电商链路}

在 Astronomy Shop 中，\texttt{NetworkDelay} 和 \texttt{PodKill} 经常把告警首先放大到 \texttt{frontend}、\texttt{checkout} 和 \texttt{payment} 等上游服务。结合\autoref{fig:dpsk-fault-alert-graph} 与\autoref{fig:dpsk-service-heatmap} 可以看出，许多样本并不是故障服务自身最先异常，而是调用方出现显著的 \texttt{HighLatency}。这类模式非常适合沉淀“高时延不一定发生在根因服务自身，应沿调用链逆向排查”的知识规则。

\subsection{Hotel Reservation：日志与错误率优先暴露依赖问题}

Hotel Reservation 的代表性现象是日志主导。大量有效样本都体现为 \texttt{ExcessiveErrorsInLogs} 与 \texttt{HighErrorRate}，而不是显著的资源或链路时延异常。结合历史记录中的原始日志可以发现，\texttt{deadline exceeded}、\texttt{dial tcp}、\texttt{CheckAvailability failed} 等错误文本经常比资源异常更早暴露问题。因此，该系统沉淀出的知识更倾向于“先看前端和依赖访问日志，再判断是网络、服务发现还是缓存/数据库问题”。

\subsection{Social Network：由上游日志反推深层下游根因}

Social Network 最具代表性的价值在于级联故障学习。许多样本中最显著的日志并不出现在真正故障点，而是先出现在 \texttt{nginx-web-server} 或 \texttt{compose-post-service} 等入口或中间层。例如 \texttt{timed out}、\texttt{Connection refused} 和 Thrift 相关异常，往往最终对应到 \texttt{post-storage-service}、\texttt{social-graph-service} 或其他更深层依赖。这类案例说明，OpsEvolver 已经能够从反复演练中积累“症状在上游、根因在下游”的微服务诊断经验。

\section{局限性分析}
\label{sec:limitations}

尽管实验数据已经显著丰富，但从结果看仍存在以下局限。
\begin{enumerate}
    \item \textbf{严格根因命中率仍偏低。} 总体严格命中率为 3.4\%，说明系统距离稳定而精确的根因定位还有明显差距。
    \item \textbf{可观测性仍受工作负载强烈影响。} Hotel Reservation 与 Social Network 的无告警样本分别达到 143 和 113 个，表明工作负载与阈值策略尚未充分放大所有故障。
    \item \textbf{知识检索与规则利用仍较轻量。} 虽然知识条目和规则持续增长，但在线阶段仍主要依赖关键词匹配与规则拼装，尚未充分利用图结构与语义检索。
    \item \textbf{多路径探索潜力尚未完全释放。} 现有实验主要基于单条主诊断轨迹，尚未系统评估多人格并行探索与结果投票的收益。
    \item \textbf{复杂组合故障覆盖仍然有限。} 本章主要覆盖单类故障演练，距离真实生产中的多故障叠加和跨系统联动场景仍有距离。
\end{enumerate}

\section{本章小结}
\label{sec:chap04_summary}

本章基于 DeepSeek-V3.2 主实验数据以及 Llama 4 Maverick 17B Instruct、Qwen3-32B 的消融实验，对 OpsEvolver 在 Astronomy Shop、Hotel Reservation 和 Social Network 三个系统上的表现进行了数据驱动分析。结果表明：系统已累计形成 959 次故障注入、703 个有效周期、971 条告警、718 条知识条目和 768 条规则；Astronomy Shop 的可观测性最强，而 Hotel Reservation 与 Social Network 中仍存在显著无告警区域；不同系统已经呈现出差异明显的故障-告警映射和知识增长路径；不同基底模型则进一步体现出“可观测覆盖、严格命中与知识沉淀密度”之间的明显权衡。总体而言，OpsEvolver 已经具备了持续积累、持续分析并暴露自身边界的研究平台特征。
