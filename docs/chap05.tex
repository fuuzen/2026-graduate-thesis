\chapter{总结与展望}
\label{cha:conclusion}

\section{工作总结}
\label{sec:summary}

OpsEvolver 围绕云原生环境下的智能运维问题，设计并实现了一个基于多智能体协作的自主演化 SRE 原型系统 OpsEvolver。与仅在静态数据集上进行根因分类的方法不同，OpsEvolver 强调让智能体进入真实微服务环境，通过受控故障注入、遥测采集、根因分析和经验沉淀构建完整闭环。在系统设计层面，OpsEvolver 围绕离线演化这一主线组织探索、诊断与反思，使故障演练、诊断推理与知识修补能够稳定联结；在实现层面，系统覆盖了容器、Pod、网络、配置和磁盘 I/O 等多个层次的故障类型，并结合指标、链路和日志三类观测信号，构建了从环境交互到诊断输出的完整链路；在知识表示层面，OpsEvolver 设计了知识条目、规则库和历史记录三类长期记忆，使系统不仅能保存最终结论，也能保存知识形成过程，从而支持诊断复盘与独立基准测评评测。

在实验分析层面，OpsEvolver 围绕应用场景、可观测性边界、告警模式、知识增长、基底模型消融、AIOpsLab 基准测评对比和代表性案例展开讨论，验证了该原型在多类微服务场景中组织完整闭环流程的可行性。

从研究结果看，OpsEvolver 的核心价值不只是“用 LLM 代替人工写一份 RCA 报告”，而是尝试把 LLM 的推理能力嵌入到持续演化的运维系统中，使其能够通过演练逐步形成贴近具体环境的知识。对于复杂且不断变化的云系统而言，这种“先演练、再沉淀、再用于后续诊断”的思路，比一次性的静态推理更符合真实运维需求。

基于 DeepSeek-V3.2 的主实验数据，OpsEvolver 已经在三个微服务系统上累计形成 959 次故障注入、703 个有效诊断周期、971 条告警、718 条知识条目和 768 条规则。其中 Astronomy Shop 的注入可观测有效率达到 100.0\%，而 Hotel Reservation 与 Social Network 中分别存在 143 个和 113 个无告警样本，说明系统已经能够同时识别“哪些故障值得学习”与“哪些故障暂时难以被观测到”。在每个系统统一截取前 300 轮的模型对比实验中，Llama 4 Maverick 17B Instruct 更有利于扩大可观测覆盖，Qwen3-32B 更倾向于高密度知识抽取，而 DeepSeek-V3.2 在 Hotel Reservation 与 Social Network 上表现出更平衡的严格命中能力。进一步地，在基于 AIOpsLab 问题集的独立基准测评上，知识增强诊断相较无知识基线在 DeepSeek-V3.2 上将正确率从 0 提升到 27.8\%，在 Qwen3-32B 上也从 39.1\% 提升到 43.5\%。虽然这些结果仍不足以支撑工业级结论，但已经足以说明：系统具备持续积累实验资产和从历史中学习的能力，论文中提出的方法论具有现实落地基础。

\section{研究不足}
\label{sec:limitations_summary}

尽管OpsEvolver 完成了较为完整的原型实现，但仍存在一些不可忽视的不足。OpsEvolver 实验目前仍以能力验证和案例分析为主，缺少更大规模、更长时间跨度以及更严格统计意义上的量化评测；严格根因命中率也仍然偏低，说明系统虽然已经能够较稳定地产生诊断轨迹，但在故障类型命名和目标组件定位上还有明显提升空间。

与此同时，知识检索模块仍采用较轻量的关键词匹配策略，尚未引入向量表示、图结构依赖或更精细的相似度学习方法；多智能体协作在原型中也偏串行化，多路径并行探索和结果投票的潜力还没有真正展开。再往前看，一些模块本身还处于原型阶段，例如链路异常阈值设计、无告警场景下的负载增强策略、安全控制粒度以及更复杂故障组合的自动生成策略，都还有进一步打磨空间。独立基准测评的结果也提示出一个现实问题，即知识增强并非在所有系统和模型上都能带来同等幅度的收益，前置的注入稳定性与可观测性仍然会直接限制最终 RCA 表现。现有系统主要聚焦“诊断与知识沉淀”，尚未深入实现自动缓解和自愈执行，因此距离真正意义上的自治 SRE 系统仍有距离。

\section{未来展望}
\label{sec:future_work}

围绕这些不足，OpsEvolver 后续仍有清晰的推进方向。首先，可以增强知识检索与表示能力，例如引入向量数据库、服务依赖图和时间序列特征，把当前较轻量的关键词检索升级为更强的语义检索与因果关联建模；对仍存在大量无告警注入的系统，则可以继续联合调整工作负载、观察窗口和阈值策略，让更多真实故障被放大为可学习样本。与此同时，故障类型本身也有必要继续扩展，在现有故障动作基础上支持组合故障、级联故障和跨命名空间依赖异常，使实验形态更接近真实生产事故。

另一条重要方向是继续完善独立评测体系。后续可以把 ITBench、AIOpsLab 等平台进一步接入统一实验脚本，形成可重复执行的量化评测流程。也可以将在跨系统迭代累积的知识库用于对 OpenRCA 这样面向不同业务拓扑的问题集进行知识增强诊断评测，用来持续检验知识增强诊断在跨系统、跨模型条件下的稳定性，进一步验证自主演化对运维效果的提升。在保证安全性的前提下，还可以进一步探索把诊断结果与回滚、扩容、重启、流量切换等缓解动作结合起来，并引入更细粒度的权限控制、动作审批与结果验证机制，逐步探索从“会诊断”走向“会处置”的演进路径。

\section{结语}
\label{sec:final_words}

总体而言，OpsEvolver 完成了一个面向云原生场景的自主演化 SRE 原型系统设计与实现。尽管距离真正可大规模生产部署的自治运维系统仍有距离，但OpsEvolver 已经证明：通过把混沌工程、外部知识库、多智能体协作和反馈学习结合起来，可以把 LLM 的推理能力嵌入到持续积累、持续复盘、持续修正的运维闭环中。OpsEvolver 的探索为后续构建更可靠、更智能、更具环境适配能力的运维系统奠定了基础。
